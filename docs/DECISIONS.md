**D33 — Synonyms are query-time, file-based, and sit AFTER the stemmer** (2026-06-13)
Synonym rules live in services/search/synonyms.txt (Solr format, versioned), parsed into a synonym_graph filter that exists only in english_search — the search_analyzer; documents are indexed without synonyms. Editing rules therefore never reindexes the 43k corpus, which is the feasibility argument for step 9's editor-managed hot reload. The filter sits after the stemmer because ES analyzes rule text through the preceding filters: rules and query tokens both arrive stemmed ("sofas" still hits "sofa, couch"). Integration tests run against the real shipped file, not fixtures. Rejected: index-time synonyms (every rule edit = full rebuild; bad rules baked in), synonyms_path inside the container (deploy coupling), placing the filter before the stemmer (plural queries silently skip expansion — no error, lost recall).

**D34 — Step-3 query: best_fields + boosts (name^3, product_class^2) + fuzziness AUTO** (2026-06-13)
Product names are short self-contained phrases, so the best single field should win; cross_fields is for values spread across fields AND cannot combine with fuzziness (ES rejects it) — two independent reasons. AUTO scales the edit budget by term length (0/1/2 for 1-2/3-5/6+ chars) so "vellvet" finds velvet but "rug" cannot mutate. Boosts are declared guesses pending step-6 measurement. Honest-test rule learned twice this step: stemming/fuzziness tests must use tokens that CANNOT match for the wrong reason ("headboards" not "beds"; single-token "vellvet" not "querry chair"). Rejected: cross_fields (incompatible + wrong data shape), fixed fuzziness 2 (short-word chaos), operator AND (recall cliff before the lab can measure it).

**D35 — Did-you-mean: phrase suggester over unstemmed name.trigram shingles; suggestion is an optional contract field** (2026-06-13)
name.trigram indexes names as 1-3-word shingles, lowercase only — suggestions are surface words for humans, never stems. Phrase suggester (gram_size 3, direct_generator suggest_mode always, default confidence) rides on every search request: no second round trip, only speaks when the rewrite scores above the typed query. SearchResponse.suggestion is optional-when-absent (never "" or null) so the lite-mode fallback stays honest without special cases. Rejected: term suggester (word-by-word suggests "platform bee"), running suggest as a separate request (extra hop for nothing), always-present nullable field (forces every consumer to handle present-but-empty).

**D36 — Autocomplete: search_as_you_type + bool_prefix, no fuzziness, same-origin proxy with 800ms cap** (2026-06-13)
name.sayt auto-generates _2gram/_3gram/prefix companions; multi_match bool_prefix treats the last word as a prefix and earlier words as complete ("fabric so" finds "three-seat fabric sofa"). No fuzziness on this path: candidate explosion per keystroke, and correcting 2-3 typed letters is guessing. The browser calls /api/autocomplete (Next route handler) which proxies the search service with AbortSignal.timeout(800) — mirrors the D12 circuit-breaker budget — and degrades to an empty suggestion list when the PC sleeps; the combobox goes quiet, /search owns honesty. Rejected: completion suggester (start-anchored only, curated inputs to maintain), hand-rolled edge n-grams (SAYT automates exactly that), browser → tunnel directly (URL leak, CORS, no degrade point).

**D37 — Highlights render by splitting on <mark>, never innerHTML; combobox is the tested ARIA pattern** (2026-06-13)
ES inserts highlight tags into raw UNESCAPED source text, so dangerouslySetInnerHTML on engine output is a stored-XSS hole: a 20-line unit-tested splitter renders segments as React text inside real <mark> elements. The header search is the WAI-ARIA combobox contract (role/aria-expanded/aria-controls/aria-activedescendant, focus stays in the input) with 200ms debounce and AbortController against the stale-response race — all asserted via RTL through the ARIA attributes themselves (apps/web gained its Vitest+jsdom+RTL rig this step, per the TESTING.md matrix). URL is state: the input follows ?q=. Rejected: sanitizer library (parsing arms race vs two known markers), focus-moving listbox (jumpy for screen readers), fetch-per-keystroke (cost + flickering stale results).

**D38 — Facets are category-scoped and aggregation-driven, modelled on JYSK** (2026-06-13)
Researched JYSK's live filters with Playwright (80 subcategory pages): a universal spine (price, colour, stock, deal flag) plus category-specific facets, with a live result count beside every value and zero-match values hidden — i.e. dynamic Elasticsearch aggregations recomputed per filter state, config-driven by category, not hardcoded per page. NORDHEM copies that architecture (not JYSK's exact inventory facets). Step 4 ships the universal spine — category, colour, material, price — with counts from terms/range aggregations; per-category attribute facets are a later refinement. Rejected: a single global facet set (JYSK proves filters differ by category, and our own data confirms the keys differ — color vs primarycolor, material vs framematerial); four generic fixed filters (teaches less than the dynamic-aggregation pattern the role is about).

**D39 — Query context vs filter context: cross-cutting filters in bool.filter, multi-select in post_filter** (2026-06-13)
The multi_match stays in bool.must (query context: it drives the BM25 score, since relevance should reflect the typed words). Category and price are cross-cutting single-select narrowers in bool.filter (filter context: unscored, cached, and they narrow every facet's counts before aggregation). Colour and material are multi-select, so their selections go in post_filter (applied after aggregations) — a colour selection narrows the returned hits without zeroing the colour facet's own counts ("tick white, still see black (1)"). Documented trade-off accepted for Step 4: post_filter facets don't narrow the OTHER facets' counts; exact cross-facet counts under simultaneous multi-selects would need per-facet filter sub-aggregations (a labelled follow-up). A regression test pins that a filter never changes relevance scores.

**D40 — Facet attributes derived from WANDS features at index time; fixed price bands; honest gaps** (2026-06-13)
A WANDS feasibility scan drove the facet list: colour/material are extracted from the pipe-delimited product_features blob (color → primarycolor fallback; primarymaterial → material → frame/upholstery fallbacks), lowercased, indexed as keyword fields on the shop documents (dynamic:strict, so the mapping grew with the doc shape). Price uses fixed shopper-friendly bands (under-500/500-1000/1000-2000/2000-plus) via a range aggregation rather than a histogram — our prices are synthetic/deterministic (D29), so bands read cleanly and give hand-countable test fixtures. Honest gaps vs JYSK, left out or labelled: no numeric firmness scale (WANDS has only mattress comfortlevel), dimensions ship in imperial inches (would need conversion + range engineering), and price itself is synthetic. Colour high-cardinality bucketing to ~12 swatches is a noted follow-up.

**D41 — Facet/sort/page state is URL-synced via pure helpers; price band is one ?price= param** (2026-06-13)
All filter, sort and page state lives in the querystring (shareable, back/forward-friendly, works before JS — extends D37). The seam is three pure functions (toggleListParam/setSingleParam/goToPage) unit-tested in apps/web; changing any filter resets to page 1, pagination preserves everything. A selected price band travels as a single ?price=500-1000 param (reuses the tested single-value helper and keeps the URL clean) and is mapped to cents bounds server-side via the search service's exported priceBandBounds, so the aggregation buckets and the filter agree by construction. Rejected: client-built min/max pairs in the URL (two params to keep in sync, duplicates the band table on the client).

**D42 — Auth: Better Auth over our own Postgres, schema generated then ported to house style; email+password always, Google env-gated** (2026-06-13)
Users live in our own Postgres (the "retailer owns its customer data" pattern) via Better Auth's Drizzle adapter over the shared db() singleton. The user/session/account/verification tables were read from Better Auth's own getAuthTables() rather than hand-guessed, then ported into schema.ts keeping snake_case DB columns and the camelCase JS keys the adapter reads (so no drift, and the schema matches the rest of the codebase); the password hash lives on account, never user. Email+password is always enabled; Google is env-gated (no GOOGLE_CLIENT_ID → the button is hidden and CI/a fresh clone need no secrets). buildAuth(db) takes the db explicitly so integration tests drive the exact app config against a real Testcontainers Postgres. Login/signup are Server Actions (not the browser client) so the forms work before and without hydration; nextCookies sets the session cookie and the guest cart is folded in by the returned user id. Rejected: rolling our own auth (sessions/hashing/CSRF is a footgun); a separate auth DB (splits the customer record); hand-writing the auth tables (version drift vs the adapter).

**D43 — Cart is a DB row keyed by a cart_id cookie; guest cart merges additively on login; favorites require login** (2026-06-13)
A cart is a real DB row, not cookie-only: guests get a cart with a null userId found by an httpOnly cart_id cookie, logged-in users get one cart enforced by a nullable-unique userId (NULLs are distinct in Postgres, so infinite guests, one cart per account). Add-to-cart is a single upsert with the quantity cap computed in SQL (LEAST(qty+n, 99)) so concurrent adds can't overshoot. On login the guest cart merges into the account cart additively (sum on overlap, union otherwise — the guest deliberately added those items this session; losing them is the worst outcome), wrapped in one transaction with the guest row deleted. Favorites require login (no guest favorites table); a signed-out heart click rolls back its optimistic flip and routes to /login. The toggle is idempotent in one round-trip (DELETE ... RETURNING, then INSERT if nothing was removed). Rejected for merge: replace / keep-user-only / take-max (all can silently drop items); cookie-only cart (lost on device switch, no clean merge).

**D44 — Checkout is one atomic transaction that snapshots everything; orders never re-derive from the live catalog** (2026-06-13)
Placing an order is a single db.transaction: read the cart joined to the live catalog, snapshot name/slug/image/unit price into order_items, write the order (with a sequence-backed human-readable NDH-YYYY-NNNNNN number and frozen cents totals from the same cartTotals() the UI uses), and clear the cart — all or nothing. Order history renders only from the snapshot, never the live shop_products, so a later price or photo change can't rewrite the past (proved by a test that mutates a price after checkout and asserts the order didn't move). A beforeCommit hook models payment authorization: a throw there rolls back the whole thing, leaving zero orders and the cart intact (also tested). Rejected: storing only productId on order lines (history breaks when a product is decurated or repriced); building the order outside a transaction (a mid-flight failure could half-write an order or double-charge the cart).

**D45 — Semantic & hybrid search: local e5 embeddings, dense_vector + kNN, RRF fused in Node; hybrid is the win, semantic alone trades recall** (2026-06-14)
Embeddings are `multilingual-e5-small` run in-process via `@huggingface/transformers` (ONNX), 384-dim, mean-pooled and L2-normalized, with the e5 `query:` / `passage:` prefixes baked into `embedQuery`/`embedPassage` so no caller can pass raw text (a missing prefix degrades relevance silently, no error). Stored in a `dense_vector` field (dims 384, index true, similarity cosine; ES auto-chose `bbq_hnsw` quantization). Passage text = name + product_class + description. Hybrid does Reciprocal Rank Fusion (1/(k+rank), k=60) as a PURE, unit-tested Node function fusing the BM25 id list and the kNN id list, chosen over Elasticsearch's native `rrf` retriever so the fusion is testable and fully understood; lexical and kNN run in parallel. `/search` gains `mode=lexical|semantic|hybrid` (semantic/hybrid retrieve ranked ids then hydrate the page via mget; facets/highlight/did-you-mean stay lexical-only), surfaced as a storefront Keyword/Meaning/Hybrid toggle. Benchmark over 480 judged queries: lexical nDCG@10 0.6615 / MRR 0.8809 / recall 38.4%, semantic 0.6857 / 0.8905 / 33.7%, hybrid 0.7284 / 0.9299 / 38.5%. Key honest finding: semantic ALONE raises ranking but DROPS recall (it misses exact tokens like model names and brands), so it is a different trade, not an upgrade; hybrid keeps lexical's recall and adds semantic's ranking, and rescued 9 zero-score queries while regressing 2 (semantic alone rescued 10 but regressed 11). One-time 43k embedding batch ~42 min. Rejected: an embedding API in the hot path (key + cost + internet — D13); the native ES rrf retriever (less testable than the Node fuser); replacing lexical with semantic (the recall drop); score-sum fusion (BM25 ~18 and cosine ~0.8 are incomparable scales, so normalizing is arbitrary).

**D46 — Editor tools: synonyms (hot-reload) vs curations (query-time), change history, benchmark-before-apply** (2026-06-14)
Synonyms are structured rows (kind equivalent/oneway, terms, mapsTo, enabled, source) in a `synonym_rules` table, converted to Solr synonym_graph lines by a pure, round-trip-tested `toSolrRule`/`parseSolrRule`. They live in the query-time `english_search` analyzer, so a change HOT-RELOADS onto a live index (close -> putSettings(analysis) -> open) with NO 43k reindex; documents are indexed with `english_text` (no synonyms), which is why this is safe. The `english_synonyms` filter uses `lenient: true` after a real live-test bug: "dresser, chest of drawers, bureau" crashed the whole analyzer because the stop filter drops "of", leaving a token position gap synonym_graph rejects (one bad rule failed every rule). lenient skips the bad rule; the editor also gets a stopword warning before saving (pure validateSynonymRule shared client+server). Synonyms seeded from synonyms.txt + 9 catalog-mined groups (mined by comparing shopper vocabulary to catalog vocabulary). CURATIONS (per-query pinned/hidden ids) are deliberately NOT in the analyzer: they are read per query at search time and applied on page 1 via the pure `curateIds` (pinned-first, hidden-dropped, deduped) with mget hydration of pinned products the ranking never returned, so a saved curation is live on the next search with no reindex and no reload, the opposite end of the cost spectrum from synonyms. CRUD is Next.js Server Actions over Postgres; Apply (synonyms) hot-reloads, save (curations) is instant. A `change_log` audit table records every action (entity/action/summary/actor/when), shown in a History ledger with apply distinguished from edits. Benchmark-before-apply: `POST /synonyms/impact` reloads the benchmark index analyzer with the saved rules and evals a 120-query sample, surfacing nDCG before the rules reach the storefront; honest finding: broad equivalent synonyms can LOWER nDCG while raising recall (precision/recall tradeoff), which is exactly why you measure before shipping. Studio IA mirrors Algolia/Elastic (synonyms separate from merchandising). Rejected/deferred: boost rules as a separate editor surface (step 7's RankingConfig already exposes field/phrase/popularity boosts; per-query and click-driven boosts belong with step 15's learning loop); Elasticsearch's stored synonyms API + _reload_search_analyzers (the close/open reload is simpler and proven); applying curations inside ES via a rules retriever (the Node curateIds is testable and mode-agnostic).

**D47 — Analytics & resilience: first-party events in Postgres, judgment-driven synthetic traffic, circuit breaker + Postgres FTS lite mode** (2026-06-15)
First-party search telemetry lives in one `search_events` table in Postgres (one row per event: a 'search' row carries mode/result_count/zero_result/latency_ms, a 'click' row carries product_id/position, with a `source` of 'live' or 'synthetic'). It is written by the Next.js backend, NOT the PC search service, so it keeps recording in lite mode when the engine is offline; the storefront fires events via navigator.sendBeacon to /api/events (best-effort: validate with a shared discriminated-union zod schema, record only valid events, swallow errors, always 204), so a click's beacon survives the navigation it triggers. CTR by position is clicks(p)/impressions(p) where impressions(p) = searches that returned >= p results (a pure, unit-tested function fed by a cheap result-count distribution, not a per-position join). The simulated-traffic generator turns the WANDS judgments into honest synthetic traffic: a position-biased cascade click model (examine(p)=decay^(p-1) times a per-grade click probability) over each query's grade-ranked judged products, Zipf-weighted session counts for a realistic head/tail, a seeded mulberry32 PRNG for reproducibility, plus out-of-catalog "miss" terms for honest zero-result signal, all written with source='synthetic' and never fed back into ranking (that would be circular). Resilience makes D12 real: a circuit breaker with an 800ms timeout on the search-service fetch, opening after 2 consecutive failures, with a half-open trial after a 10s cooldown; while open the backend skips the service entirely and serves "lite mode" = Postgres full-text search over the shop catalog, labelled mode:"fallback" with an honest banner and a /status page reporting full vs lite. Lite-mode FTS uses OR semantics (rewriting plainto_tsquery's '&' output to '|', injection-safe because plainto sanitizes the input) after live testing showed AND-only returned 0 results for "oak bed" (OR now returns 176) - a fallback must favour recall. Honest degradation: lite mode has no facets, synonyms or semantic ranking, and the UI says so. Rejected: writing events from the search service (would stop logging exactly when the PC is down, when you most need to know); fetch-per-event without sendBeacon (lost on navigation); feeding synthetic clicks into ranking (circular); ES native _profile/rrf for these dashboards (Postgres already holds the events and works in lite mode); plainto AND semantics for the fallback (silent zero-result on ordinary multi-word queries).
